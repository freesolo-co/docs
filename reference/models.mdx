Flash trains a **LoRA adapter** on top of a curated catalog of base models. You
set the base model with one line in your config, and list the live catalog from
the CLI:

```bash
flash models
```

```toml
model = "Qwen/Qwen3.5-4B"   # required — see `flash models` for the options
```

`flash models` is always the source of truth. The table below mirrors the
current catalog.

## Catalog

Every catalog model supports **both SFT and GRPO**, and all are hybrid reasoning
("thinking") capable.

| Model (`model =`) | Parameters | Training class | Serving tier | Reasoning |
| --- | ---: | --- | --- | --- |
| `Qwen/Qwen3.5-0.8B` | 0.9B | RTX 4090 | L4 | Hybrid |
| `openbmb/MiniCPM5-1B` | 1.2B | RTX 4090 SFT / RTX 5090 GRPO | L4 | Hybrid |
| `Qwen/Qwen3.5-2B` | 2.3B | RTX 4090 SFT / RTX 5090 GRPO | L4 | Hybrid |
| `Qwen/Qwen3.5-4B` | 4.7B | RTX 4090 SFT / A100-80GB GRPO | L4 FP8 | Hybrid |
| `Qwen/Qwen3.5-9B` | 9.7B | RTX 5090 SFT / A100-80GB GRPO | L4 FP8 | Hybrid |
| `Qwen/Qwen3.6-35B-A3B` | 35B total / ~3B active | H200 SFT / B200 GRPO | A100-80GB FP8 | Hybrid |

## Serving prices

Serving is billed per token after deployment. Prompt and completion tokens have
separate per-model rates, and cached prompt tokens use the model's cached-input
rate. The prices below are **per 1M tokens** and include Freesolo's 1.2 serving
markup over typical provider rates; your Freesolo billing dashboard shows the
charges you actually accrue.

| Model | Prompt / 1M | Completion / 1M | Cached prompt / 1M |
| --- | ---: | ---: | ---: |
| `Qwen/Qwen3.5-0.8B` | $0.012 | $0.060 | $0.0024 |
| `openbmb/MiniCPM5-1B` | $0.012 | $0.060 | $0.0024 |
| `Qwen/Qwen3.5-2B` | $0.024 | $0.120 | $0.0048 |
| `Qwen/Qwen3.5-4B` | $0.036 | $0.180 | $0.0072 |
| `Qwen/Qwen3.5-9B` | $0.120 | $0.180 | $0.0240 |
| `Qwen/Qwen3.6-35B-A3B` | $0.180 | $1.200 | $0.0600 |

The **Cached prompt** column is the rate for prompt tokens served from the
**prefix cache**. Caching is automatic whenever a request's prompt shares a
leading prefix with a recent one (a shared system prompt, or a multi-turn chat);
only prompt tokens are cacheable, so completion tokens always bill at the full
per-token rate. See [Billing](/guides/deploy-and-chat#billing).

### Notes per model

- **`Qwen/Qwen3.5-0.8B`**: the smallest Qwen3.5. Cheapest for smoke tests and
  fast iteration.
- **`openbmb/MiniCPM5-1B`**: an on-device-class small model on a standard Llama
  architecture.
- **`Qwen/Qwen3.5-2B`**: a small, capable step up when 0.8B underfits.
- **`Qwen/Qwen3.5-4B`**: a balanced starting point for most tasks.
- **`Qwen/Qwen3.5-9B`**: the largest dense catalog model. GRPO uses an 80 GB-class
  card, so use it when quality matters more than the run cost.
- **`Qwen/Qwen3.6-35B-A3B`**: a Mixture-of-Experts checkpoint (35B total, ~3B
  active per token) and the largest model in the catalog. SFT runs on an H200
  class and GRPO runs on a B200 class.

<Note>
  The Qwen3.5 entries are **text-only** fine-tunes: the checkpoints are natively
  multimodal, but Flash trains and serves the language model only.
</Note>

## Reasoning ("thinking") models

Every catalog model is **hybrid-reasoning**: it can run with or without an
explicit reasoning step. Turn reasoning on for a run with one line in your
config:

```toml
thinking = true
```

See the [`thinking` field](/reference/configuration#top-level) in the
configuration reference.
