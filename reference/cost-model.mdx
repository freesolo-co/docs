`flash train --cost` prints a pre-flight estimate of what a run will cost before
you submit it. At its core the number is simple:

```text
total = billable training-loop hours x GPU $/hr
```

Setup and cold start time are still reported because they affect wall clock, but
they are **not billed**. The estimate is computed locally from Flash's catalog
of model sizes, GPU compute, setup constants, and GPU rate snapshots, so the
same config prices the same way. Every dollar amount and time here is an
illustrative example; your run's preview comes from `flash train --cost`, and
the GPU class snapshot comes from [`flash gpus`](/reference/cli#discovery).

<Note>
  The estimate is a preview, not the charge. A run is
  [billed when it finishes](/guides/training#cost-and-billing), for the measured
  training-loop GPU time it used. Failed and cancelled runs that never reach a
  successful terminal state are not charged for training.
</Note>

## What drives your cost

Three things matter:

- **The training loop.** For GRPO each step samples completions, grades them with
  your reward, then updates the policy; cost scales with the number and length of
  completions. For SFT each step is one forward/backward pass over a batch; cost
  scales with batch size and sequence length.
- **GPU class.** The allocator picks the cheapest validated managed GPU class
  that fits your run from the available provider pool. A faster card costs more
  per hour but may finish sooner.
- **Setup time.** Container boot, dependency install, model load, and vLLM init
  are included in wall-clock reporting and the wall-time cap, but not in the
  customer training charge.

## The master equation

Everything below expands these lines:

```text
total       = billable_hours x gpu_hourly_usd
billable    = train_seconds / 3600
wall_clock  = setup + train
setup       = cold start (boot + deps + model load, + vLLM init for GRPO)
train       = steps x per_step_seconds
per_step    = step FLOPs / (gpu_peak_flops x MFU)
```

`MFU` is model-FLOPs utilization, the fraction of a GPU's peak compute a real run
sustains. It is below 1 because LoRA on small batches never saturates the card;
the values are calibrated against observed run time.

## GPU class and price

You do not pin a GPU. The estimator sizes the run's VRAM, then prices the
cheapest fitting class in the current catalog. The submit-time allocator can
choose a different validated provider/class if availability changes.

| GPU class | VRAM | RunPod $/hr | Peak bf16 TFLOPS |
| --- | ---: | ---: | ---: |
| RTX 4090 | 24 GB | $0.69 | 165 |
| RTX 5090 | 32 GB | $0.99 | 210 |
| A100 PCIe | 80 GB | $1.39 | 312 |
| A100 SXM | 80 GB | $1.49 | 312 |
| RTX Pro 6000 | 96 GB | $2.09 | 250 |
| H100 | 80 GB | $3.29 | 990 |
| H200 | 141 GB | $4.39 | 990 |
| B200 | 180 GB | $5.89 | 2250 |

Run [`flash gpus`](/reference/cli#discovery) for the CLI's current GPU table.
Submitted runs record the realized GPU and provider in `flash runs` and
`flash status`.

<Warning>
  **The estimate is a local forecast.** It prices the cheapest fitting class from
  the local snapshot. The allocator provisions from the validated provider pool,
  so the card actually used can differ from the estimate.
</Warning>

## Setup (cold start)

Setup is fixed wall time before the first optimizer step. It is reported, but it
is not part of billable training time:

```text
setup = boot(120s) + deps(90s) + model_load   (+ vLLM init for GRPO)

  model_load = 235s + download_gb / 0.4 GB/s   # fixed init, then the download
  download_gb = params_b x 2                    # bf16 weights, 2 bytes/param
  vLLM init  = 120s                             # GRPO only
```

| Component | Value | Covers |
| --- | --- | --- |
| Container boot | 120 s | image pull + start |
| Dependency install | 90 s | resolving and installing worker deps |
| Model load (base) | 235 s | checkpoint deserialize, GPU placement, CUDA init |
| Model download | `params_b x 2 / 0.4` s | pulling the bf16 checkpoint from the hub |
| vLLM init | 120 s (GRPO only) | spinning up the rollout engine |

<Tip>
  Setup can dominate a short run's wall clock, even though it is not billed. The
  cheapest way to iterate quickly is still to start with a smaller base model and
  a short run.
</Tip>

## Per-step time

One step's wall time is its compute divided by the GPU's effective throughput,
`per_step = FLOPs / (peak x MFU)`.

| Phase | FLOPs / token / param | MFU |
| --- | ---: | ---: |
| SFT forward + backward | 6 | 0.25 |
| GRPO rollout (decode) | 2 | 0.12 |
| GRPO policy update | 8 | 0.35 |

Decode has the lowest MFU because autoregressive generation is
memory-bandwidth-bound, not compute-bound. The factors come from counting passes:
a forward is 2, a backward is 4, and the GRPO update is a policy
forward+backward (6) plus a frozen-reference forward (2).

<CodeGroup>

```text SFT
per_step = (6 x P x batch_size x seq_len) / (peak x 0.25)

  P    = model parameters
  peak = GPU peak bf16 FLOP/s
```

```text GRPO
per_step = gen + reward + update

  N    = batch_size x group_size      # completions sampled this step
  D    = N x completion_len           # total generated tokens (max_tokens each)

  gen     = (2 x P x D) / (peak x 0.12)   # vLLM rollout
  update  = (8 x P x D) / (peak x 0.35)   # policy + frozen-reference update
  reward  = ceil(N / 16) x latency        # grading, 16 concurrent slots
```

</CodeGroup>

<Tip>
  Reward grading runs concurrently in 16 slots, so its wall time is the number
  of waves, `ceil(N / 16)`, not the completion count `N`. `latency` is the
  reward seconds per completion (default 1.0 s; graders range from fast regex to
  slower LLM judges).
</Tip>

## Steps and the wall cap

Steps turn per-step time into the training total, and a wall cap bounds the
whole job:

- **Steps (GRPO).** `train = steps x per_step`, where `steps` is
  your `train.steps`.
- **Steps (SFT).** SFT has no `train.steps`. The estimator derives the step count
  from the dataset as `epochs x ceil(num_examples / batch)`, capped by
  `max_steps`, where `batch` is the run's realized effective batch (32 by
  default, unless you set `batch_size`). `num_examples` is `max_examples` if you
  pin it, otherwise the real environment size, which the estimator loads your
  environment locally to count.
- **Wall cap.** A run is bounded by a 24h wall-clock cap (a platform default; see
  [GPU & infrastructure](/reference/configuration#gpu-infrastructure-managed)).
  Setup consumes wall-clock budget first; only the training-loop time that
  actually runs is billable.

## Worked example

Take a GRPO run on the smallest catalog model with explicit knobs:

```toml
model = "Qwen/Qwen3.5-0.8B"   # ~0.9B params (from flash models)
algorithm = "grpo"

[train]
steps = 100
batch_size = 8
group_size = 8
max_tokens = 512
```

For this config, the estimator chooses an **RTX 4090** (24 GB, peak 165 TFLOPS)
at `$0.69/hr`. Then `P = 0.9e9`, `peak = 165e12`, `N = 8 x 8 = 64`
completions, and `D = 64 x 512 = 32,768` generated tokens.

**Per step:**

```text
gen      = (2 x 0.9e9 x 32768) / (165e12 x 0.12) = 2.979 s
update   = (8 x 0.9e9 x 32768) / (165e12 x 0.35) = 4.087 s
reward   = ceil(64 / 16) x 1.0 = 4 x 1.0         = 4.000 s
per_step = 2.979 + 4.087 + 4.000                 = 11.06 s
```

**Setup** (GRPO, so vLLM init applies; download = 0.9 x 2 = 1.8 GB):

```text
model_load = 235 + 1.8 / 0.4 = 239.5 s
setup      = 120 + 90 + 239.5 + 120 = 569.5 s = 9.5 min
```

**Total:**

```text
train    = 100 x 11.06 s = 1106 s = 18.4 min
wall     = 569.5 + 1106  = 1675.5 s = 0.47 h
billable = 1106 / 3600   = 0.31 h
total    = 0.31 h x $0.69/hr = $0.21
```

`flash train --cost` prints the same distinction between wall clock and billable
time:

```text
Run        : Qwen/Qwen3.5-0.8B  [GRPO, 100 steps]
GPU        : RTX 4090 on auto (24 GB; run needs >= 24 GB) @ $0.69/hr
Setup      : 9.5 min (cold start: boot + deps + model load + vLLM init; not billed)
Per step   : 11.06 s
Train      : 18.4 min
Wall clock : 0.47 h
Billable   : 0.31 h (training only)
TOTAL      : $0.21
```

## How to make a run cheaper

Each lever maps back to a term above:

- **A smaller base model.** Smaller `P` cuts every per-step term and shrinks the
  cold-start download. See [Supported models](/reference/models).
- **Fewer or shorter completions (GRPO).** `group_size` and `max_tokens` both
  scale generation and the policy update linearly through `D`.
- **Fewer steps.** `train` is linear in `steps`.
- **A faster card** can be cheaper overall when it finishes enough sooner to
  offset a higher `$/hr`, but you do not choose this; the allocator picks the
  cheapest fitting class.

## Why your bill can differ from the estimate

The estimate is a deterministic forecast; the final charge is measured. They can
diverge because:

- **The bill is measured, not estimated.** You are charged for the actual
  training-loop GPU time the successful run used, not the setup time.
- **A different card may be provisioned.** The estimate prices the cheapest
  fitting class; the allocator provisions from the validated pool, which can
  change with provider availability.
- **Pricing can drift.** The estimate uses a GPU `$/hr` snapshot; the final run
  uses the platform rate in effect when the run completed.
- **MFU and setup are calibrated approximations.** Real utilization, boot time,
  model load, and dependency install time vary with the model, environment, and
  provider.
- **Reward latency is an estimate.** GRPO uses a default per-completion grading
  latency unless your run overrides it; a slow [reward](/guides/environments)
  costs more wall time than the default assumes.
