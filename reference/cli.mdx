The CLI is `flash` (installed from the **`freesolo-flash`** package, see the
[Quickstart](/quickstart#1-install-the-cli)). Every run-lifecycle command is a
thin, authenticated call to the hosted control plane. Run `flash --help` or
`flash <command> --help` for inline help.

## Global flags

| Flag | Effect |
| --- | --- |
| `-V`, `--version` | Print the Flash version and exit |
| `--debug` | Show full tracebacks on error (otherwise errors print one clean line) |
| `-v`, `-vv` | Increase log verbosity (`-v` info, `-vv` debug) |

## Auth & identity

<ResponseField name="flash login" type="[--api-key KEY] [--freesolo-url URL] [--api-url URL]">
  Log in with your Freesolo API key. The key is verified against Freesolo and
  saved locally. Defaults: `--api-key` from `FREESOLO_API_KEY`; `--freesolo-url`
  from `FREESOLO_BASE_URL` (else `https://api.freesolo.co`); `--api-url` from
  `FLASH_API_URL`.
</ResponseField>

<ResponseField name="flash whoami">
  Show the identity your stored key resolves to.
</ResponseField>

<ResponseField name="flash version">
  Print the Flash version.
</ResponseField>

## Discovery

<ResponseField name="flash models">
  List supported base models with their algorithms, quantization, and reasoning
  capability.
</ResponseField>

<ResponseField name="flash gpus">
  List managed GPU classes with VRAM, live per-provider `$/hr`, and where each
  class is validated.
</ResponseField>

## Environments

<ResponseField name="flash env setup">
  Scaffold `environments/` and `configs/` in the current directory with a
  starter environment plus `configs/grpo.toml`, `configs/sft.toml`, and
  `configs/endpoints.toml`.
</ResponseField>

<ResponseField name="flash env init NAME">
  Scaffold a new local Freesolo environment at `environments/NAME/`.
</ResponseField>

<ResponseField name="flash env push --name NAME [PATH]">
  Publish a local Freesolo environment to Freesolo's managed Environments Hub
  (private) and print its id (`your-name/NAME`). `--name` is required; `PATH`
  defaults to the current directory. Prefer passing an environment folder so
  helper modules and data sidecars are uploaded with `environment.py`. Fully
  managed with your Freesolo API key.
</ResponseField>

<ResponseField name="flash env install ENV_ID">
  Install a published Freesolo environment locally for authoring and `--dry-run`.
</ResponseField>

<ResponseField name="flash env list">
  List installed Hub environments and local environments you can publish.
</ResponseField>

## Training

<ResponseField name="flash train CONFIG [flags]">
  Submit a managed training run from a TOML config and follow its logs.
  Submitting charges the pre-flight cost estimate to your org and is gated on
  balance (see [Cost & billing](/guides/training#cost-and-billing)).

  | Flag | Effect |
  | --- | --- |
  | `--dry-run` | Validate the config locally and print the resolved job spec; don't submit |
  | `--cost` | Print the pre-flight USD cost estimate for the config and exit (no submit) |
  | `--background` | Submit and return immediately instead of following logs |
  | `--set KEY=VALUE` | Override a config value (dotted key, repeatable) |
  | `--config FILE` | Deep-merge additional TOML for config composition (repeatable) |
</ResponseField>

## Run management

<ResponseField name="flash ps">
  List your runs with state, accrued cost, GPU, and model.
</ResponseField>

<ResponseField name="flash status RUN_ID">
  Print a run's full status JSON.
</ResponseField>

<ResponseField name="flash attach RUN_ID">
  Re-follow a running job's logs to completion (resumable any time).
</ResponseField>

<ResponseField name="flash logs RUN_ID [-f]">
  Print a run's logs. `-f`/`--follow` streams new lines until the run ends.
</ResponseField>

<ResponseField name="flash cost RUN_ID">
  Show a run's accrued cost in USD.
</ResponseField>

<ResponseField name="flash cancel RUN_ID">
  Cancel a run (best-effort).
</ResponseField>

## Serving

<ResponseField name="flash deploy RUN_ID [flags]">
  Register a trained adapter with the managed multi-LoRA serving service.
  Serving is [billed per token](/guides/deploy-and-chat#billing); there is no
  per-day idle charge.

  | Flag | Default | Effect |
  | --- | --- | --- |
  | `--dry-run` | - | Preview the deploy without creating it |
</ResponseField>

<ResponseField name="flash chat RUN_ID -m MESSAGE [flags]">
  Send a message to a deployed adapter and print the reply.

  | Flag | Default | Effect |
  | --- | --- | --- |
  | `-m`, `--message` | (required) | The user message |
  | `--max-tokens` | `512` | Max tokens to generate |
  | `--temperature` | `0.0` | Sampling temperature |
</ResponseField>

<ResponseField name="flash deployments">
  List active serving deployments with GPU and endpoint.
</ResponseField>

<ResponseField name="flash undeploy RUN_ID">
  Deregister a run's adapter from serving so it no longer accepts requests.
</ResponseField>

## MCP server (for coding agents)

Flash also ships a stdio [MCP](https://modelcontextprotocol.io) bridge so coding
agents can drive the platform. It isn't a `flash` subcommand; launch it with:

```bash
python -m flash.mcp.server
```

It authenticates with the same stored credentials as the CLI (`flash login`) and
exposes tools to list models, submit and monitor training runs, and deploy
adapters.
