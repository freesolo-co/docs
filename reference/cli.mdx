The CLI is `flash`. Run `flash --help` or `flash <command> --help` for
inline help.

## Global flags

| Flag | Effect |
| --- | --- |
| `-V`, `--version` | Print the Flash version and exit |
| `--debug` | Show full tracebacks on error (otherwise errors print one clean line) |
| `-v`, `-vv` | Increase log verbosity (`-v` info, `-vv` debug) |

## Auth & identity

<ResponseField name="flash login" type="[--api-key KEY] [--freesolo-url URL] [--api-url URL]">
  Log in with your Freesolo API key. The key is verified against Freesolo and
  saved locally, and the resolved identity is printed on success. Defaults:
  `--api-key` from `FREESOLO_API_KEY`; `--freesolo-url` from `FREESOLO_BASE_URL`
  (else `https://api.freesolo.co`); `--api-url` from `FLASH_API_URL`.
</ResponseField>

<ResponseField name="flash whoami">
  Show the identity your stored key resolves to.
</ResponseField>

<ResponseField name="flash version">
  Print the Flash version.
</ResponseField>

## Discovery

<ResponseField name="flash models">
  List the supported base model ids and their parameter sizes. See
  [Supported models](/reference/models) for algorithms, reasoning, and pricing.
</ResponseField>

<ResponseField name="flash gpus">
  List managed RunPod GPU classes with VRAM and a static `$/hr` snapshot.
</ResponseField>

## Environments

<ResponseField name="flash env setup">
  Scaffold a starter project into the current directory: `environment.py`, a tiny
  `datasets/train.jsonl`, `configs/rl.toml` (GRPO) plus `configs/sft.toml` (SFT),
  and a `TRAINING.md` playbook for coding agents. Files that already exist are left
  untouched, so it is safe to rerun.
</ResponseField>

<ResponseField name="flash env push --name NAME [PATH]">
  Publish a local Freesolo environment to Freesolo's managed Environments Hub
  (private) and print its id (`your-name/NAME`). `--name` is required; `PATH`
  defaults to the current directory. Pass `.` or any folder with
  `environment.py` at its root so helper modules, `datasets/`, and common
  sibling data files are uploaded.
</ResponseField>

<ResponseField name="flash env install ENV_ID">
  Install a published Freesolo environment locally for authoring and `--dry-run`.
</ResponseField>

<ResponseField name="flash env list">
  List installed Hub environments and local environments you can publish.
</ResponseField>

## Training

<ResponseField name="flash train CONFIG [flags]">
  Submit a managed training run from a TOML config and follow its logs.
  A run is billed only when it finishes successfully, for the GPU time it
  used (see [Cost & billing](/guides/training#cost-and-billing)); the
  pre-flight `--cost` estimate is derived in
  [How cost is computed](/reference/cost-model).

  | Flag | Effect |
  | --- | --- |
  | `--dry-run` | Validate the config locally and print the resolved job spec; don't submit |
  | `--cost` | Print the pre-flight USD cost for the config and exit (no submit) |
  | `--background` | Submit and return immediately instead of following logs |
  | `--set KEY=VALUE` | Override a config value (dotted key, repeatable) |
  | `--config FILE` | Deep-merge additional TOML for config composition (repeatable) |
</ResponseField>

## Run management

<ResponseField name="flash runs">
  List your runs with state, algorithm, accrued cost, GPU, and model.
</ResponseField>

<ResponseField name="flash status RUN_ID [--logs] [-f]">
  Print a run's full status JSON, including accrued cost. `--logs` prints current
  logs before the status JSON. `-f`/`--follow` streams logs until the run ends,
  then prints final status.
</ResponseField>

<ResponseField name="flash cancel RUN_ID">
  Cancel a run.
</ResponseField>

## Serving

<ResponseField name="flash deploy RUN_ID [flags]">
  Register a trained adapter with the managed multi-LoRA serving service.
  Serving is [billed per token](/guides/deploy-and-chat#billing); there is no
  per-day idle charge.

  | Flag | Default | Effect |
  | --- | --- | --- |
  | `--step` | (final adapter) | Deploy a specific [checkpoint step](/guides/deploy-and-chat#deploy-a-specific-checkpoint) instead of the run's final adapter |
  | `--dry-run` | - | Preview the deploy without creating it |
</ResponseField>

<ResponseField name="flash checkpoints RUN_ID">
  List a run's per-step checkpoints that are available to deploy — including
  intermediate checkpoints from a run that stopped mid-RL. Serve one with
  `flash deploy RUN_ID --step N`.
</ResponseField>

<ResponseField name="flash chat RUN_ID -m MESSAGE [flags]">
  Send a message to a deployed adapter and stream the reply as it is generated.

  | Flag | Default | Effect |
  | --- | --- | --- |
  | `-m`, `--message` | (required) | The user message |
  | `--max-tokens` | `512` | Max tokens to generate |
  | `--temperature` | `0.0` | Sampling temperature |
</ResponseField>

<ResponseField name="flash deployments">
  List active serving deployments with GPU and endpoint.
</ResponseField>

<ResponseField name="flash undeploy RUN_ID">
  Deregister a run's adapter from serving so it no longer accepts requests.
</ResponseField>

## MCP server (for coding agents)

Flash also ships a stdio [MCP](https://modelcontextprotocol.io) bridge. It isn't
a `flash` subcommand; launch it with:

```bash
python -m flash.mcp.server
```

It authenticates with the same stored credentials as the CLI (`flash login`) and
exposes tools to list models, submit and monitor training runs, and deploy
adapters: `list_models`, `create_training_run`, `get_run_status`,
`get_run_logs`, and `deploy_adapter`.
