Fixes for the issues you're most likely to hit, grouped by where they show up.
Most `flash` errors print one clean line; add `--debug` to any command for the
full traceback.

## Installation & CLI

<AccordionGroup>
  <Accordion title="flash: command not found">
    The CLI installs a single `flash` command, but its install location has to be
    on your `PATH`.

    - If you installed with `uv tool install freesolo-flash`, make sure uv's tool
      bin directory is on your `PATH` (run `uv tool update-shell`, then restart
      your shell).
    - Confirm the install with `flash version`.

  </Accordion>

  <Accordion title="I installed `flash` but it's the wrong tool">
    The CLI is published to PyPI as **`freesolo-flash`**, the bare `flash` name
    belongs to an unrelated project. Reinstall the right one:

    ```bash
    uv tool install freesolo-flash
    ```

  </Accordion>
</AccordionGroup>

## Authentication

<AccordionGroup>
  <Accordion title="flash login fails or commands report 401 / invalid key">
    Every command authenticates with a **Freesolo API key**, verified against
    Freesolo at login.

    - Create a key in your dashboard at [freesolo.co](https://freesolo.co).
    - Log in once: `flash login --api-key <your-key>` (or set `FREESOLO_API_KEY`
      instead of passing `--api-key`).
    - Confirm who the stored key resolves to: `flash whoami`.

  </Accordion>

  <Accordion title="Pointing at a non-default Freesolo deployment">
    By default the CLI talks to `https://api.freesolo.co`. To target a different
    deployment, set `--freesolo-url` (or `FREESOLO_BASE_URL`) at login.
  </Accordion>
</AccordionGroup>

## Environments

<AccordionGroup>
  <Accordion title="flash env push fails">
    Make sure the folder you push contains an `environment.py` file with a
    `load_environment()` function that returns a Freesolo environment, then:

    ```bash
    flash env push --name math math
    ```

    It prints the published id (`your-name/math`) to put in your config's
    `[environment] id`.

  </Accordion>

  <Accordion title="ModuleNotFoundError: No module named 'freesolo' (local authoring)">
    Training is managed, so the env's dependencies live server-side. But local
    authoring needs them installed locally, as does any `--dry-run`/`--cost`
    that **loads** the environment (for example, to count examples for SFT
    cost):

    ```bash
    uv pip install freesolo
    ```

    To pull a published env into your project for local work, use
    `flash env install your-name/your-env`.

  </Accordion>

<Accordion title="Environment won't import / name collision">
  If your environment module shares a name with an installed Python package, it
  can shadow or be shadowed by that package. Keep helper module names distinct
  from installed packages.
</Accordion>

  <Accordion title="What id do I reference in my config?">
    The `[environment] id` must be a published Freesolo environment id, produced
    by `flash env push`, for example `your-name/your-env`.
    A local file path is not a valid id, so publish it first or install an
    existing one.
  </Accordion>
</AccordionGroup>

## Configuration

<AccordionGroup>
  <Accordion title="unsupported model '...'">
    `model` must be one of the ids in the curated catalog. List the valid ids:

    ```bash
    flash models
    ```

    Managed runs train catalog models only. Need one that isn't listed?
    [Let us know](https://freesolo.co/contact). See
    [Supported models](/reference/models).

  </Accordion>

  <Accordion title="Unknown config key or section rejected">
    Flash rejects unknown config sections and `[train]` keys **at parse time**, so
    a typo fails fast instead of silently training against defaults. Check the key
    against the [configuration reference](/reference/configuration) and validate
    locally:

    ```bash
    flash train config.toml --dry-run
    ```

  </Accordion>

<Accordion title="My [gpu] table or hf_repo is being ignored">
  GPU **class** selection is fully managed and `hf_repo` is assigned for you, so
  a pinned `gpu.type` or `hf_repo` is ignored. The allocator picks the cheapest
  fitting GPU, and each run streams artifacts into a private Hugging Face repo
  provisioned server-side. Remove `gpu.type` and `hf_repo` from your config. See
  [GPU & infrastructure](/reference/configuration#gpu-infrastructure-managed).
</Accordion>

  <Accordion title="unsupported algorithm">
    `algorithm` must be `grpo` (the default) or `sft`. Fix the value and
    re-validate with `--dry-run`.
  </Accordion>
</AccordionGroup>

## GPUs & fit

<AccordionGroup>
  <Accordion title="GRPO needs a bigger GPU than SFT for the same model">
    Expected. GRPO has to generate sample answers (rollouts) as it trains, so it
    runs a generation engine alongside training that holds a second copy of the
    weights plus a KV cache. That needs more VRAM than SFT. The 9B model, for
    instance, fits a 48 GB card for SFT but needs an 80 GB-class card for GRPO.
    Pick a smaller model or use `sft`. See [Supported models](/reference/models).
  </Accordion>
</AccordionGroup>

## Training runs

<AccordionGroup>
  <Accordion title="When am I charged for a run?">
    A run is billed only when it **finishes successfully**, for the actual GPU
    time it used; a run that doesn't complete isn't billed. Nothing is charged at
    submit. Preview the cost before you submit with the pre-flight estimate:

    ```bash
    flash train config.toml --cost
    ```

  </Accordion>

<Accordion title="GRPO reward is stuck at 0">
  The task is too hard for the model at its current ability: GRPO learns by
  rewarding the model's better attempts (rollouts), so if none ever score,
  there's nothing to reinforce. Try a stronger/larger base model,
  make the task easier to get started, or double-check your reward actually
  returns a positive reward for good answers.
</Accordion>

<Accordion title="Reward is stuck at 1 (or never moves)">
  Usually the reward is not discriminative: it scores almost everything the
  same. Make the reward function separate better answers from worse ones so GRPO
  has a spread of scores to learn from. See [Environments](/guides/environments).
</Accordion>

  <Accordion title="Did pressing Ctrl-C kill my run?">
    No. `Ctrl-C` during `flash train` just **detaches** you; the run keeps going
    server-side. Re-follow it any time, and cancel explicitly if you mean to:

    ```bash
    flash runs              # state and cost of your runs
    flash status <run-id> -f    # re-follow the logs
    flash cancel <run-id>       # stop it
    ```

  </Accordion>

  <Accordion title="A GPU hiccup or unavailability interrupted training">
    Runs are supervised server-side: a stall watchdog plus bounded auto-retry that
    **resumes from the last streamed checkpoint**, and the allocator walks to the
    next-cheapest validated GPU class if one is unavailable. A transient failure
    usually doesn't cost you the whole run. Watch with `flash status <run-id> -f`.
  </Accordion>
</AccordionGroup>

## Serving

<AccordionGroup>
  <Accordion title="Deploy or the first request is slow">
    Deploy registers and warms the adapter on Freesolo's managed serving
    service. Large models can take a few minutes before the endpoint is ready.
    See [Deploy & chat](/guides/deploy-and-chat).
  </Accordion>

  <Accordion title="Am I being charged while the deployment is idle?">
    No. Serving is **billed per token** for the requests you actually send; there
    is no per-day idle or warm-GPU charge. `flash undeploy <run-id>` deregisters
    the adapter entirely.
  </Accordion>

  <Accordion title="Calling the endpoint from my own code is rejected">
    Deployments are OpenAI-compatible. Use the endpoint from `flash deployments`
    as the `base_url`, your **Freesolo API key** as the `api_key`, and the
    `<run-id>` as the `model`. See
    [Use it from your own code](/guides/deploy-and-chat#use-it-from-your-own-code).
  </Accordion>
</AccordionGroup>

## Getting help

Still stuck? Add `--debug` to surface the full traceback, then reach out through
[freesolo.co/contact](https://freesolo.co/contact) with the run id (`flash runs`)
and the failing command.
