Improvement in Freesolo is not a one-time fine-tuning run. It is a structured, continuously operating flywheel in which live product feedback drives model refinement, which drives better product behavior, which generates richer feedback for the next cycle. Each iteration compounds on the last: the evaluation sets get more representative, the model gets more precisely tuned to your feature's requirements, and the infrastructure investment you made in the first cycle pays dividends across every AI feature you build afterward.

## The Improvement Flywheel

The flywheel runs in three interconnected loops. In the innermost loop, instrumented production traces are evaluated against your feature's success criteria and failures are identified at the step level. In the middle loop, those failure signals inform targeted model updates — SFT labels concentrated on hard cases, RL reward signals calibrated to the failure patterns, APO candidates screened against the same evaluation pipeline. In the outer loop, improved models ship back to production, generate new instrumented traces, and the cycle restarts with better data.

What makes this a flywheel rather than a linear pipeline is that each cycle improves the quality of the next. Better-performing models produce fewer failure traces, which makes the remaining failures easier to analyze and fix. More representative evaluation sets catch regressions earlier. Fine-tuned adapters from previous features provide useful initialization for new ones. The system gets smarter about how to get smarter.

## Supervised Fine-Tuning: The Cold Start

When Freesolo begins improving a new feature, it does not immediately apply Reinforcement Learning. RL requires a model that already produces outputs in roughly the right space — a model that is so far from the target distribution that most of its outputs fail hard checks will generate reward signals that are too sparse to be useful for training.

Supervised Fine-Tuning (SFT) solves the cold-start problem. Freesolo uses labeled traces — instrumented examples where the correct output is known or can be constructed — to directly teach the model the output format, the reasoning style, and the domain-specific patterns your feature requires. SFT does not optimize for a reward signal; it optimizes for imitation of demonstrated behavior. That makes it fast, stable, and highly effective at establishing a starting point from which RL can operate productively.

In practice, SFT concentrates on the failure modes surfaced during evaluation: the schema violations, the empty result sets, the cases where the model's output was plausible but wrong. Labeled examples in these areas give RL a better-initialized model to work with, which means RL training converges faster and reaches higher performance.

Freesolo runs SFT on managed GPU clusters, using LoRA (Low-Rank Adaptation) adapters rather than full parameter updates. LoRA adapters are computationally efficient, modular, and composable — you can apply a feature-specific adapter on top of a base model without retraining the base, and you can stack adapters for different features independently.

## Reinforcement Learning: Reward-Guided Refinement

Once SFT has established a viable starting point, Freesolo applies Reinforcement Learning to push the model toward its full potential on your feature. RL optimizes for a reward signal rather than imitating demonstrated behavior, which means it can discover solutions that go beyond what the labeled training data explicitly shows — including novel ways to satisfy the feature's success criteria that no human labeler would have thought to demonstrate.

Freesolo's reward signals combine two components designed to capture the full quality picture:

**Quantity signals** answer the question: did the model produce a usable result at all? Execution success, schema compliance, and result-set size all feed into the quantity dimension. A model that passes all hard checks on a given input earns a positive quantity signal. A model that fails — returning an empty result, a malformed schema, or an unexecutable output — earns a negative one. Quantity signals are fast to compute and provide dense feedback across the training distribution.

**Quality signals** answer the question: was the result actually good? LLM-as-judge scores, calibrated to your product's evaluation rubric, provide the quality dimension. Quality signals are more expensive to compute than quantity signals, but they capture the distinction between outputs that are technically valid and outputs that are genuinely useful. Together, quantity and quality reward signals give the RL training loop a complete picture of what it should be optimizing toward.

Per-step win rate — the fraction of inputs on which the model produces a passing output at each step of the pipeline — is tracked throughout RL training, giving you a continuous, interpretable view of where improvement is happening and where work remains.

## Automatic Prompt Optimization: Treating the Prompt as a Parameter

Even after SFT and RL have refined the model's weights, significant performance gains remain available through prompt optimization. In most AI systems, prompts are written by hand, iterated manually, and changed infrequently. Freesolo treats the prompt as a tunable parameter — something to be systematically optimized using the same evaluation signals that guide model training.

Freesolo's Automatic Prompt Optimization (APO) follows the DSPy-style approach: candidate prompt variants are generated, each variant is scored against the evaluation pipeline using the same hard checks and LLM-as-judge rubrics used during RL, and only variants that improve aggregate performance across the evaluation set are retained. Variants that perform worse — even on a subset of inputs — are discarded. This ensures that prompt changes are always net-positive: they help on the cases they're designed to help, and they don't regress anything.

APO runs continuously as part of the Improve flywheel, which means your prompts stay optimized as your instrumentation data grows and your evaluation sets become more representative. You don't need to manually re-examine and rewrite prompts every time your feature evolves — APO handles that systematically.

<Note>
APO operates under a strict guardrail: it cannot re-introduce behaviors that RL training has already corrected. Prompt variants are screened against the full evaluation suite before adoption, which means any variant that would restore a failure mode that RL eliminated — even a failure mode that the prompt variant was not designed to address — is automatically rejected. This constraint ensures that the improvement flywheel only moves in one direction: forward. Gains made in the RL phase are treated as a floor, and APO is only permitted to raise it.
</Note>

## How Training Compounds Across Features

One of the most durable advantages of the Freesolo Engine is that improvement work compounds across features, not just within them. LoRA adapters trained on one feature encode product-specific behaviors — response formats, domain vocabulary, reasoning patterns — that transfer to related features. Evaluation sets built from one feature's instrumented queries often contain useful signals for adjacent features. Prompt optimization techniques that work on one feature's reward landscape tend to generalize.

This compounding effect means that the second AI feature you build with Freesolo is faster and cheaper to optimize than the first, and the tenth is faster and cheaper than the second. Each deployment cycle contributes to a proprietary layer of product-specific AI capability that becomes a durable competitive asset.
