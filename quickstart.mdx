title: "Quickstart"
description: "Install the CLI, log in, and train, deploy, and chat with your first model."
---

This walks you from zero to a deployed, chatting model. It takes a few minutes
of your time plus the training run itself.

<Note>
  You need a **Freesolo API key**. Create one in your dashboard at
  [freesolo.co](https://freesolo.co). It's how every `flash` command
  authenticates. No GPU or provider credentials required.
</Note>

## 1. Install the CLI

The CLI is published to PyPI as **`freesolo-flash`** (the bare `flash` name is
taken by another project). It installs a single command, `flash`.

<CodeGroup>

```bash uv
uv tool install freesolo-flash
```

```bash pipx
pipx install freesolo-flash
```

```bash pip
pip install freesolo-flash
```

</CodeGroup>

Verify it:

```bash
flash version
```

## 2. Log in

```bash
flash login --api-key <your-freesolo-key>
```

Flash verifies the key against Freesolo and stores it locally, so you only do
this once. You can also set `FREESOLO_API_KEY` instead of passing `--api-key`.

```bash
flash whoami    # confirm who your stored key resolves to
```

## 3. Scaffold a project

```bash
flash env setup
```

This drops a ready-to-edit starter in the current directory:

```
environment.py            # a Freesolo environment (task + reward)
configs/
  grpo.toml               # a GRPO training config to start from
  sft.toml                # an SFT training config to start from
  endpoints.toml          # somewhere to stash deploy endpoints
```

<Note>
  `flash env setup` writes `environment.py` in the current folder. Publish a
  folder, not just the Python file, so helper modules and datasets beside it are
  included.
</Note>

## 4. Publish your environment

An [environment](/guides/environments) defines what your model trains on and
how it's graded. Publish the scaffolded one to the managed Environments Hub to get an id:

```bash
flash env push --name starter .
```

This prints an environment id of the form `your-name/starter`.

<Note>
  Publishing is fully managed. `flash env push --name ... .` packages the
  environment from the current folder and uploads it to Freesolo's hosted
  Environments Hub
  (published **private**) using just your Freesolo API key. Already have a
  published env id? Skip straight to step 5.
</Note>

## 5. Configure your run

Open `configs/grpo.toml` and set the one thing that's yours, the
environment id from the previous step:

```toml configs/grpo.toml
model = "Qwen/Qwen3.5-4B"
algorithm = "grpo"

[environment]
id = "your-name/starter"

[train]
steps = 150
lora_rank = 32
seeds = [0]
```

<Note>
  The only fields you must set are `model` and `[environment] id`. **The GPU and
  the run's artifact storage are fully managed**: the allocator picks the
  cheapest GPU class that fits across providers, and each run gets its own private
  HF repo automatically. You don't pick a GPU or supply a Hugging Face repo.
</Note>

Validate it locally first. `--dry-run` parses and checks the config with no
credentials, server, or GPU:

```bash
flash train configs/grpo.toml --dry-run
```

See what the run will cost before committing. `--cost` prints the pre-flight
USD cost and exits without submitting:

```bash
flash train configs/grpo.toml --cost
```

<Note>
  **Training pricing is GPU-time pricing, not token pricing.** Flash calculates
  the wall-clock time for your selected model, algorithm, and training settings,
  multiplies that by the managed GPU price, and shows the pre-flight charge
  with `--cost`. `flash train configs/grpo.toml` charges that cost to your
  org balance up front before any GPU starts; serving token charges are separate
  and only begin after `flash deploy`.
</Note>

## 6. Train

```bash
flash train configs/grpo.toml
```

Submitting charges the pre-flight cost (the `--cost` number) to your org's
balance up front; if the balance can't cover it, the run is rejected and never
starts. Flash then follows the logs live. Press `Ctrl-C` to detach. The run
keeps going on the server, and you can follow it again any time:

```bash
flash ps                  # list your runs and their state/cost
flash status <run-id>     # status JSON, including accrued cost
flash status <run-id> -f  # stream logs until the run finishes
```

The run reaches `done` when training finishes.

## 7. Deploy

Serve the trained adapter on Freesolo's managed serving service. There is no
per-day idle charge: deployment registers the adapter, and serving is billed per
prompt and completion token for requests you send:

```bash
flash deploy <run-id>
```

## 8. Chat

```bash
flash chat <run-id> -m "Hello! What can you do?"
```

That's a full loop. When you're done, tear the endpoint down:

```bash
flash undeploy <run-id>
```
