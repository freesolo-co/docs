This walks you from zero to a deployed, chatting model. It takes a few minutes
of your time plus the training run itself.

<Note>
  You need a **Freesolo API key**. Create one at
  [freesolo.co/sign-in](https://freesolo.co/sign-in) — it's how every `flash`
  command authenticates. No GPU or provider credentials required.
</Note>

## 1. Install the CLI

The CLI is published to PyPI as **`freesolo-flash`** (the bare `flash` name is
taken by another project). It installs a single command, `flash`.

<CodeGroup>

```bash uv
uv tool install freesolo-flash
```

```bash pipx
pipx install freesolo-flash
```

```bash pip
pip install freesolo-flash
```

</CodeGroup>

Verify it:

```bash
flash version
```

## 2. Log in

```bash
flash login --api-key <your-freesolo-key>
```

Flash verifies the key against Freesolo and stores it locally, so you only do
this once. You can also set `FREESOLO_API_KEY` instead of passing `--api-key`.

```bash
flash whoami    # confirm who your stored key resolves to
```

## 3. Scaffold a project

```bash
flash env setup
```

This drops a ready-to-edit starter in the current directory:

```
environment.py            # a Freesolo environment (the task + rubric)
datasets/
  train.jsonl             # the starter dataset (one {"input","output"} per line)
configs/
  rl.toml                 # a GRPO training config to start from
  sft.toml                # an SFT training config to start from
TRAINING.md               # a playbook for iterating on the run
```

<Note>
  `flash env setup` writes the starter as a single project in the current
  directory: `environment.py` plus a `datasets/` folder. You publish the whole
  folder with `flash env push --name <name> .` (see step 4).
</Note>

## 4. Publish your environment

An [environment](/guides/environments) defines what your model trains on and
how it's graded. Publish the scaffolded project to get an id:

```bash
flash env push --name my-env .
```

This prints an environment id of the form `owner/name`.

<Note>
  Publishing is fully managed — `flash env push` packages your project folder
  (`environment.py` plus its `datasets/`) and uploads it to Freesolo's hosted
  Environments Hub (published **private**) using just your Freesolo API key.
  Already have a published env id? Skip straight to step 5.
</Note>

## 5. Configure your run

Open `configs/rl.toml` and set the one thing that's yours — the environment id
from the previous step:

```toml configs/rl.toml
model = "Qwen/Qwen3.5-4B"
algorithm = "grpo"

[environment]
id = "your-name/my-env"             # from `flash env push --name my-env .`

[train]
steps = 150
lora_rank = 32
```

<Note>
  The only fields you must set are `model` and `[environment] id`. **The GPU and
  the run's artifact storage are fully managed** — the allocator picks the
  cheapest GPU class that fits (RunPod, with Lambda Labs as a capacity
  fallback), and each run gets its own private HF repo automatically. You don't
  pick a GPU or supply a Hugging Face repo.
</Note>

Validate it locally first — `--dry-run` parses and checks the config with no
credentials, server, or GPU:

```bash
flash train configs/rl.toml --dry-run
```

See what the run will cost before committing — `--cost` prints the pre-flight
USD estimate and exits without submitting:

```bash
flash train configs/rl.toml --cost
```

## 6. Train

```bash
flash train configs/rl.toml
```

Flash follows the logs live. The run is billed when it finishes, on its realized
cost (the `--cost` number above is a pre-flight estimate, not the final charge).
Press `Ctrl-C` to detach — the run keeps going on the server, and you can
re-attach any time:

```bash
flash runs                       # list your runs and their state/cost
flash status <run-id> --follow   # re-follow the logs to completion
flash status <run-id> --logs     # print the run's logs
```

The run reaches `done` when training finishes.

## 7. Deploy

Serve the trained adapter on Freesolo's managed serving service. It
**scales to zero when idle** (no charge for idle time) and cold-starts on the
first request; you're billed per token for the requests it serves:

```bash
flash deploy <run-id>
```

## 8. Chat

```bash
flash chat <run-id> -m "Hello! What can you do?"
```

That's a full loop. When you're done, tear the endpoint down:

```bash
flash undeploy <run-id>
```
