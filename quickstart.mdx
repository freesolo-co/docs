title: "Quickstart"
description: "Install the CLI, log in, and train, deploy, and chat with your first model."
---

This walks you from zero to a deployed, chatting model. It takes a few minutes
of your time plus the training run itself.

<Note>
  You need a **Freesolo API key**. Create one in your dashboard at
  [freesolo.co](https://freesolo.co) — it's how every `flash` command
  authenticates. No GPU or provider credentials required.
</Note>

## 1. Install the CLI

The CLI is published to PyPI as `flash` and installs a single command, `flash`.

<CodeGroup>

```bash uv
uv tool install flash
```

```bash pipx
pipx install flash
```

```bash pip
pip install flash
```

</CodeGroup>

Verify it:

```bash
flash version
```

## 2. Log in

```bash
flash login --api-key <your-freesolo-key>
```

Flash verifies the key against Freesolo and stores it locally, so you only do
this once. You can also set `FREESOLO_API_KEY` instead of passing `--api-key`.

```bash
flash whoami    # confirm who your stored key resolves to
```

## 3. Scaffold a project

```bash
flash lab setup
```

This drops a ready-to-edit starter in the current directory:

```
environments/
  starter_env.py          # a verifiers environment (the task + rubric)
configs/
  verifiers_grpo.toml     # a training config to start from
  endpoints.toml          # somewhere to stash deploy endpoints
```

<Note>
  `flash lab setup` writes a single starter module, `environments/starter_env.py`.
  To scaffold an additional, named environment later, use
  [`flash env init NAME`](/guides/environments), which creates a per-task
  directory with a snake_cased module file instead (e.g. `flash env init my-task`
  → `environments/my-task/my_task.py`). Both forms are valid env sources you can
  `flash env push`.
</Note>

## 4. Publish your environment

An [environment](/guides/environments) defines what your model trains on and
how it's graded. Publish the scaffolded one to the Prime Hub to get an id:

```bash
flash env push environments/starter_env.py
```

This prints an environment id of the form `owner/name`.

<Note>
  Publishing is fully managed — `flash env push` packages your env and uploads it
  to Freesolo's hosted Environments Hub (published **private**) using just your
  Freesolo API key. **No Prime Intellect account or `prime` CLI needed.** Already
  have a published env id? Skip straight to step 5.
</Note>

## 5. Configure your run

Open `configs/verifiers_grpo.toml` and set two things — the environment id from
the previous step, and an HF dataset repo to stream adapters/checkpoints into:

```toml configs/verifiers_grpo.toml
model = "Qwen/Qwen3.5-4B"
algorithm = "grpo"

[environment]
id = "your-name/starter-env"        # from `flash env push`

[train]
hf_repo = "your-org/your-runs"      # HF dataset repo for adapters/checkpoints
steps = 150
lora_rank = 32
seeds = [0]

[gpu]
type = "RTX 5090"                   # or omit to auto-pick the cheapest fit
```

Validate it locally first — `--dry-run` parses and checks the config with no
credentials, server, or GPU:

```bash
flash train configs/verifiers_grpo.toml --dry-run
```

## 6. Train

```bash
flash train configs/verifiers_grpo.toml
```

Flash submits the run and follows the logs live. Press `Ctrl-C` to detach — the
run keeps going on the server, and you can re-attach any time:

```bash
flash ps                  # list your runs and their state/cost
flash attach <run-id>     # re-follow the logs
flash logs <run-id> -f    # stream just the logs
```

The run reaches `done` when training finishes.

## 7. Deploy

Serve the trained adapter. The default mode is **scale-to-zero** — it costs
nothing while idle and cold-starts on the first request:

```bash
flash deploy <run-id>
```

## 8. Chat

```bash
flash chat <run-id> -m "Hello! What can you do?"
```

That's a full loop. When you're done, tear the endpoint down:

```bash
flash undeploy <run-id>
```
