title: "Quickstart"
description: "Install the CLI, log in, and train, deploy, and chat with your first model."
---

This quickstart takes you from an empty directory to a deployed model you can
chat with. It takes a few minutes of your time plus the training run itself. By
the end you will have trained a LoRA adapter on a managed GPU, served it, and
called it from the CLI.

## Before you begin

Make sure you have:

- A **Freesolo API key**. Create one in your dashboard at
  [freesolo.co](https://freesolo.co). It's how every `flash` command
  authenticates. No GPU or provider credentials required.
- **Python with `uv`, `pipx`, or `pip`** to install the CLI.
- A **terminal** open in an empty directory to scaffold your project into.

<Note>
  This guide uses the scaffolded starter environment so you can finish a full
  loop fast. To write your own task and reward, see
  [Environments](/guides/environments); for SFT vs GRPO and tuning, see
  [Training](/guides/training); for the serving API, see [Deploy &
  chat](/guides/deploy-and-chat). For the concepts behind each step, see [How
  Flash works](/how-flash-works).
</Note>

## Step 1: Install the CLI

<CodeGroup>

```bash uv
uv tool install freesolo-flash
```

```bash pipx
pipx install freesolo-flash
```

```bash pip
pip install freesolo-flash
```

</CodeGroup>

Verify it:

```bash
flash version
```

## Step 2: Log in

```bash
flash login --api-key <your-freesolo-key>
```

Flash verifies the key against Freesolo and stores it locally, so you only do
this once. You can also set `FREESOLO_API_KEY` instead of passing `--api-key`.

```bash
flash whoami    # confirm who your stored key resolves to
```

## Step 3: Scaffold a project

```bash
flash env setup
```

This drops a ready-to-edit starter in the current directory:

```
environment.py            # a Freesolo environment (task + reward)
configs/
  grpo.toml               # a GRPO training config to start from
  sft.toml                # an SFT training config to start from
```

## Step 4: Publish your environment

An [environment](/guides/environments) defines
[what your model trains on](/guides/datasets) and how it's graded.
[Publish](/guides/environments#publish-it) the scaffolded one to the managed
Environments Hub to get an id:

```bash
flash env push --name starter .
```

This prints an environment id of the form `your-name/starter`.

## Step 5: Configure and validate your run

Open `configs/grpo.toml` and set the one thing that's yours, the environment id
from the previous step:

```toml configs/grpo.toml
model = "Qwen/Qwen3.5-4B"
algorithm = "grpo"

[environment]
id = "your-name/starter"

[train]
steps = 150
lora_rank = 32
seeds = [0]
```

[Validate it locally](/guides/training#validate-before-you-submit) first.
`--dry-run` parses and checks the config with no credentials, server, or GPU:

```bash
flash train configs/grpo.toml --dry-run
```

See what [the run will cost](/guides/training#cost-and-billing) before
committing. `--cost` prints the pre-flight USD cost and exits without submitting:

```bash
flash train configs/grpo.toml --cost
```

## Step 6: Train

```bash
flash train configs/grpo.toml
```

<Note>
  This is the first step that spends money. Submitting charges the pre-flight
  cost (the `--cost` number) to your org's balance up front; if the balance
  can't cover it, the run is rejected and never starts.
</Note>

Flash then follows the logs live. Press `Ctrl-C` to detach. The run keeps going
on the server, and you can
[follow it again any time](/guides/training#monitor-a-run):

```bash
flash ps                  # list your runs and their state/cost
flash status <run-id>     # status JSON, including accrued cost
flash status <run-id> -f  # stream logs until the run finishes
```

The run reaches `done` when training finishes.

## Step 7: Deploy

Serve the trained adapter on Freesolo's
[managed serving service](/guides/deploy-and-chat). There is no per-day idle
charge: deployment registers the adapter, and serving is
[billed per prompt and completion token](/guides/deploy-and-chat#billing) for
requests you send:

```bash
flash deploy <run-id>
```

## Step 8: Chat

```bash
flash chat <run-id> -m "Hello! What can you do?"
```

That's a full loop. When you're done,
[tear the endpoint down](/guides/deploy-and-chat#manage-deployments):

```bash
flash undeploy <run-id>
```

## Essential commands

The commands you used above, plus the ones you'll reach for next. Run
`flash <command> --help` for the full set of flags, or see the
[CLI reference](/reference/cli).

| Command              | What it does                               | Example                                |
| -------------------- | ------------------------------------------ | -------------------------------------- |
| `flash login`        | Store your Freesolo API key locally        | `flash login --api-key <key>`          |
| `flash env setup`    | Scaffold a starter environment and configs | `flash env setup`                      |
| `flash env push`     | Publish an environment and print its id    | `flash env push --name starter .`      |
| `flash train`        | Submit a run and follow its logs           | `flash train configs/grpo.toml`        |
| `flash train --cost` | Preview the pre-flight cost, no submit     | `flash train configs/grpo.toml --cost` |
| `flash ps`           | List your runs with state, cost, GPU       | `flash ps`                             |
| `flash status`       | Show run status, or stream its logs        | `flash status <run-id> -f`             |
| `flash deploy`       | Serve a trained adapter                    | `flash deploy <run-id>`                |
| `flash chat`         | Send a message to a deployment             | `flash chat <run-id> -m "hi"`          |
| `flash undeploy`     | Tear a deployment down                     | `flash undeploy <run-id>`              |

## Pro tips

<AccordionGroup>
  <Accordion title="Validate, then price, before you train" icon="circle-check">
    Run `flash train configs/grpo.toml --dry-run` to catch a typo with no
    credentials or GPU, then `--cost` to see the price, before you submit and
    spend.
  </Accordion>
  <Accordion title="Detach freely; the run keeps going" icon="circle-pause">
    `Ctrl-C` during `flash train` only stops following the logs. The run
    continues server-side. Follow it again any time with
    `flash status <run-id> -f`.
  </Accordion>
  <Accordion title="Already have an environment? Skip authoring" icon="forward">
    If you have a published env id (yours or one shared with you), put it in your
    config's `[environment] id` and jump straight to step 5. No need to scaffold
    or publish.
  </Accordion>
  <Accordion title="Start small, then scale only the steps" icon="gauge-simple">
    Finish one short run end to end before you scale up. When you do, raise
    `train.steps` and change little else, so a longer run reproduces a shape you
    have already seen.
  </Accordion>
</AccordionGroup>
