title: "Deploy & chat"
description: "Serve a trained adapter and talk to it over an OpenAI-compatible API."
---

Once a run reaches `done`, deploy its adapter and chat with it — from the CLI or
any OpenAI-compatible client.

`flash deploy` registers your adapter with Freesolo's **managed multi-LoRA
serving service**. Every adapter for a given base model is served from one shared
GPU that **scales to zero when idle**, so there's nothing to keep warm and **no
per-day idle charge** — you're billed per token only for the requests you send
(see [Billing](#billing)).

## Deploy

```bash
flash deploy <run-id>
```

That's it — there are no serving modes or warm-up settings to choose. The adapter
is registered, scales to zero when idle, and cold-starts on the first request.

Preview what a deploy would do without creating it:

```bash
flash deploy <run-id> --dry-run
```

### Deploy an intermediate checkpoint

A GRPO run streams a deployable checkpoint at each save interval. List them and
deploy a specific step instead of the final adapter — this works even for a run
you cancelled mid-training:

```bash
flash checkpoints <run-id>          # list deployable per-step checkpoints
flash deploy <run-id> --step 120    # serve the checkpoint at step 120
```

(SFT-only runs produce a single final adapter and have no intermediate
checkpoints.)

## Billing

Serving is **billed per token**. Each request costs its prompt + completion
tokens at the base model's serverless list price plus a 20% markup, debited from
your org's prepaid balance. There is no separate per-day, idle, or warm-GPU
charge — a deployment that sits idle costs nothing; you pay only for the tokens of
the requests it actually serves.

## Chat from the CLI

```bash
flash chat <run-id> -m "Summarize the plot of Hamlet in two sentences."
```

Options:

| Flag              | Default    | Effect                     |
| ----------------- | ---------- | -------------------------- |
| `-m`, `--message` | (required) | The user message to send   |
| `--max-tokens`    | 512        | Maximum tokens to generate |
| `--temperature`   | 0.0        | Sampling temperature       |

```bash
flash chat <run-id> -m "Write a haiku about GPUs" --temperature 0.8 --max-tokens 128
```

## Manage deployments

```bash
flash deployments        # list active deployments: run id, GPU, endpoint
flash undeploy <run-id>  # deregister the adapter from serving
```

<Note>
  Because serving scales to zero, there's no idle cost to stop. `flash undeploy`
  deregisters the adapter so it no longer accepts requests; you're only ever
  billed per token for requests you actually send.
</Note>

## Use it from your own code

Deployments are OpenAI-compatible, so you can point any OpenAI SDK at the
endpoint and use your Freesolo API key:

```python
from openai import OpenAI

client = OpenAI(
    base_url="<your-endpoint>/v1",   # endpoint from `flash deployments` (already includes https://)
    api_key="<your-freesolo-key>",
)

resp = client.chat.completions.create(
    model="<run-id>",
    messages=[{"role": "user", "content": "Hello!"}],
)
print(resp.choices[0].message.content)
```

## Take the adapter with you

Want the trained weights outside Freesolo's serving? Export the adapter to a
Hugging Face repo you own:

```bash
flash export --adapter-id <run-id> --repository your-name/your-adapter
```

Pass `--step <N>` to export a specific checkpoint, and `--public` to create the
destination repo public (it's private by default). Authentication uses your
`HF_TOKEN` (from your shell or a local `.env`), or pass `--api-key <hf-token>`.
